**Canonical results** vs **demo artifacts**:

1. Purpose
- Canonical: production-style persistent datasets for downstream consumers.
- Demo artifacts: convenience snapshots/previews for walkthrough/debug.

2. Location
- Canonical: `data/processed/dq_results`, `audit_log`, `universe_membership`, etc.
- Demo artifacts: `data/processed/layer*_..._demo/...` (for example `layer6_false_alarm_demo/...`).

3. Schema/coverage
- Canonical: full run output schema, partitioned tables, all rows for the run scope.
- Demo artifacts: selected outputs (preview tables, summaries, manifests), often row-limited (`head(...)`).

4. Pipeline owner
- Canonical: written by `RunEngine`.
- Demo artifacts: written by `run_demo_*` scripts.

5. Stability
- Canonical: intended as the standard source for UI/analytics/audit.
- Demo artifacts: intended for testing/explaining; structure can vary by demo script.

6. UI usage
- Preferred: point UI to canonical `data/processed/dq_results`.
- Demo-only mode: point UI to specific demo files when canonical tables are not built.
